# Проект: SteamBalance

**Категория:** FinTech / Digital Services Aggregator  
**Тип:** High-load сервис пополнения цифровых балансов  
**Модель:** B2C + интеграции с внешними платёжными и сервисными провайдерами

---

# 1. Бизнес-контекст

SteamBalance — сервис пополнения баланса игровых платформ:

- Steam
    
- PUBG: Battlegrounds
    
- и другие цифровые сервисы
    

Формально продукт — это «пополнение баланса».  
По сути — это финтех-посредник, агрегирующий платёжные шлюзы и поставщиков цифровых услуг.

Сервис не выполняет пополнение напрямую, а выступает оркестратором транзакции между:

1. Клиентом
    
2. Платёжной системой
    
3. Поставщиком услуги пополнения
    

---

# 2. Основной бизнес-процесс

## Общий flow

1. Пользователь вводит:
    
    - логин Steam
        
    - сумму
        
    - выбирает способ оплаты
        
2. Создаётся Order
    
3. Пользователь переходит в платёжную систему
    
4. От платёжного провайдера приходит webhook (success / fail)
    
5. При успешной оплате:
    
    - сервис инициирует запрос к поставщику пополнения
        
6. Получает статус выполнения
    
7. Обновляет финальный статус заказа
    

---

# 3. Архитектурная модель

## Основные доменные сущности

### 1. Orders

Хранят:

- user identifier (Steam login)
    
- amount
    
- payment_id
    
- provider_id
    
- статус
    
- timestamps
    

### Возможные статусы:

- created
    
- waiting_payment
    
- paid
    
- processing
    
- success
    
- failed
    
- refunded
    

---

### 2. Payments

- способ оплаты (карта, СБП и др.)
    
- статус транзакции
    
- provider_payment_id
    
- webhook_payload (для аудита)
    

---

### 3. Provider Requests

- provider_id
    
- external_request_id
    
- статус выполнения
    
- связь с order
    

---

# 4. Интеграции

Минимум две внешние интеграции:

## 1. Платёжный провайдер

- редирект пользователя
    
- асинхронный webhook
    
- возможные повторные уведомления
    
- необходимость idempotency
    

## 2. Поставщик пополнения Steam

- API-запрос на пополнение
    
- polling / callback
    
- обработка ошибок
    

---

## Важные технические требования

- idempotent webhook обработка
    
- retry-механизм
    
- защита от race-condition
    
- аудит транзакций
    
- корректная финализация заказа
    

---

# 5. Работа с большими объёмами данных

В проектировочной задаче фигурировала таблица на **500+ млн строк**.

Это предполагает:

- очень большой объём заказов
    
- долгоживущий сервис
    
- необходимость оптимизации запросов
    

Используются:

- keyset pagination (WHERE id > X)
    
- отказ от OFFSET при больших объёмах
    
- индексация по status / created_at / id

# 6. Технологический стек

## Backend

- Laravel
    
- Eloquent (Active Record)
    
- частично raw SQL
    

## База данных

- MySQL
    

## Очереди

- Redis
    
- собственные демоны-воркеры
    
- планируется переход на Laravel queues
    

## WebSocket

- библиотека: Swool
    
- используется для:
    
    - realtime-обновления цен
        
    - отображения онлайна
        
    - уведомлений
        

## Frontend

- Next.js
    
- админка — нативный JS
    
- backend предоставляет REST API

# 8. Нагрузочные и операционные риски

Проект работает в условиях:

- периодических блокировок провайдеров
    
- ограничений со стороны Cloudflare
    
- регуляторных ограничений (РКН)
    
- нестабильных платёжных шлюзов
    

Это требует:

- fallback-провайдеров
    
- быстрой смены интеграций
    
- ручного мониторинга
    
- высокой адаптивности архитектуры
