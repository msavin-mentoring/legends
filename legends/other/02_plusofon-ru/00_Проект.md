## Плюсофон

### Проект: DID API + платформа IP-телефонии и аналитики CDR (Caller ID / «этикетки» / МНП / антиспам / блокировки)

#### Описание системы

Платформа управления телефонной номерной ёмкостью и аналитики звонков для задач IP-телефонии: хранение и обработка детализации звонков (**CDR**), обогащение справочниками операторов и внутренними данными, построение аналитики и «этикеток» звонящих (caller labels), а также формирование действий по номерам (например, ограничение/блокировка) с синхронизацией в смежные контуры.

Система решает две большие задачи:

1. **DID API (управление номерами/услугами/интеграциями)** — доменная логика вокруг номерной ёмкости и интеграционных сценариев.
2. **CDR-контур (ингест + enrichment + аналитика)** — конвейер загрузки звонков, дозаполнение атрибутов (операторы/агенты), вычисление признаков подозрительной активности и подготовка агрегатов/выгрузок.

#### Ключевые термины домена

* **CDR (Call Detail Record)** — факт звонка (время, номер A/номер B, длительность, статус/коды разъединения и пр.). Базовая сущность для аналитики/контроля качества/антиспама.
* **Этикетки (caller labels)** — метки/категории, сопоставляемые номеру (системные или пользовательские), используемые в отчётности и идентификации.
* **МНП (Mobile Number Portability)** — перенос номера между операторами; требует определения *актуального* оператора номера (часто на дату звонка).
* **БДПН** — источник/контур перенесённых номеров для определения текущего оператора при МНП.
* **NPDB** — микросервис/справочник нумерации/операторов, используемый в пайплайне обогащения и синхронизациях.
* **MAV** — отдельный микросервис/контур для передачи изменений по номерам и результатов аналитики (в т.ч. suspected calls).
* **XSIM** — контур, где применяются действия по номерам (ограничения/блокировки и т.п.).
* **DUL** — внешний контур/API, по решениям/статусам которого формируются действия и далее синхронизируются в XSIM.
* **Agents / Numbers** — связка «номер → агент → организация», нужна для обогащения CDR и агрегации статистики по агентам.
* **ЕПВВ / исторические сведения** — контур/таблицы исторических совпадений/верификации (поиск/выгрузка исторических вызовов по спискам номеров).

#### Основной поток данных

1. **Ingestion CDR**
   Вызовы вычитываются из источника телефонии/CDR-контура за заданное окно времени, нормализуются (формат номера, валидность), сохраняются в PostgreSQL.

2. **Обогащение операторами (МНП)**
   Для записей без заполненных операторов выполняется enrichment по номерам A/B: батчи запросов в BDPN/NPDB API, ретраи, контроль конкурентности и rate limiting. Результаты дозаписываются обратно без затирания уже заполненных значений.

3. **Обогащение агентами**
   Для номеров A/B определяется агент через таблицы `numbers → agents`, чтобы строить аналитику и агрегаты по агентам/организациям.

4. **Агрегации и антиспам-аналитика**
   Регламентные задачи строят статистику (почасовая/суточная/помесячная), рассчитывают признаки подозрительной активности (например, высокая повторяемость звонков на один и тот же номер), формируют витрины/таблицы под отчёты.

5. **Формирование действий и синхронизация контуров**
   По правилам, статусам и ответам DUL формируются списки номеров для действий и выполняется синхронизация изменений в XSIM и MAV.

#### Архитектура

**Backend и сервисы**

* Основной backend: **PHP 8.3 + Laravel** (DID API, бизнес-логика, интеграции, административные сценарии, планировщик задач).
* Микросервисы/контуры: минимум выделялись **NPDB** и **MAV**.
* Высоконагруженные джобы/ETL: **Go** (оптимизация по времени и снижение давления на PostgreSQL).
  Типовые утилиты:

  * `fill` — перенос/загрузка звонков за окно времени (ingestion);
  * `enrich` — дозаполнение операторов по номерам A/B через BDPN/NPDB;
  * вспомогательные команды (`stamp`, `seed-integration`, `server`) как инфраструктурные утилиты.

**Базы данных**

* **PostgreSQL** — основная БД.
* CDR-таблица (`cdr` / `calls`) **партиционирована по дням**: `PARTITION BY RANGE(call_timestamp)`, партиции вида `calls_YYYY_MM_DD`.
* Дополнительные таблицы/витрины:

  * `cdr_operators` — справочник операторов (в т.ч. коды BDPN);
  * `cdr_stats_monthly` — помесячные агрегаты с уникальностью на `(month, a_number, agent_id, operator_id)` для идемпотентных прогонов;
  * `mav_suspected_calls` — подозрительные вызовы с уникальностью на `(run_id, a_number, b_number, call_timestamp)`;
  * `historical_call_hits` — исторические совпадения/выгрузки (индексация под поиск по `(a_number, call_timestamp)`).

**Фоновые задачи**

* **cron** для Go-джоб.
* **Laravel Scheduler** для регламентных процедур, синхронизаций и обслуживающих задач.
* Брокер очередей как обязательный элемент не использовался в предоставленных данных: часть асинхронности/оркестрации решалась через БД и расписания.

**Интеграции и импорт**

* BDPN/NPDB API: батчи (типично ~50, в отдельных режимах больше), ретраи (до ~5), rate limiting, ограничение конкурентности.
* MAV / XSIM / DUL: интеграции по API и местами через прямое подключение к БД (по контексту проекта).
* Импорт **CSV** (сторонний сервис выгружал файлы в mount-директорию; далее фильтрация и загрузка в таблицы).
* Хранилище документов “Эльза/Эльба” (S3-подобный object storage).
* Работа с контуром исторических сведений (ЕПВВ).

**Практики обработки больших объёмов**

* В Go-`fill` использовалась **keyset pagination** (`ORDER BY call_timestamp, id` + курсор `(call_timestamp, id)`), без `OFFSET`.
* Индексация и обслуживание партиций без длительных блокировок:

  * индексы на партициях **CONCURRENTLY** (например, `(number_a, call_timestamp)`),
  * SQL/plpgsql-функции под агрегации и сервисные операции на диапазоне партиций.

#### Нагрузка и данные

* **Объём CDR**: ~**1 000 000 записей/сутки**.
* **Партиции**: ~**400 дневных партиций**, ~**1 000 000 записей** в каждой ⇒ суммарно порядка **~400 млн CDR** (оценка по указанным цифрам).
* **RPS на интеграции**: в отдельных интеграционных сценариях учитывались rate limit’ы порядка **~300 RPS** (иногда лимитирование временно отключали для прогонов/разгрузки очередей задач).
* **Пайплайн ingestion — сравнение реализаций по времени**:

  * PHP реализация: ~**2 часа**,
  * «чисто SQL в PostgreSQL»: ~**13 минут**, но с чрезмерной нагрузкой на БД,
  * Go реализация: ~**3 минуты** при контролируемой нагрузке на PostgreSQL.

#### Моя зона ответственности

Backend-разработка и эксплуатационно-ориентированная оптимизация пайплайнов обработки телеком-данных.

* Реализация и оптимизация ingestion-конвейера CDR: вычитка звонков за окно времени, нормализация/валидация, запись в PostgreSQL.
* Разработка высоконагруженных ETL/джоб на Go:

  * keyset pagination,
  * батч-upsert,
  * разделение этапов `fill` / `enrich` для управляемой нагрузки и идемпотентных прогонов.
* Обогащение CDR операторами для МНП:

  * батчирование запросов в BDPN/NPDB,
  * ретраи, ограничение конкурентности, rate limiting,
  * логика сохранения без перезаписи уже заполненных операторов.
* Обогащение CDR агентами (numbers → agents) и подготовка данных для агрегаций/отчётности.
* Регламентные джобы на Laravel Scheduler:

  * синхронизация статусов/действий по номерам между контурами,
  * каскадные обновления/удаления в нескольких таблицах,
  * прокидывание изменений в MAV и применение действий через XSIM (с учётом решений DUL).
* Импорт больших CSV (до ~**2 млн строк**) из mount-директории: фильтрация по правилам домена, загрузка в целевые таблицы.
* Аналитика/антиспам-задачи:

  * выявление повторяемости «номер → один и тот же номер» как сигнал подозрительной активности,
  * агрегации входящих/исходящих «этикеток» по часам/месяцам с группировкой по оператору и агенту,
  * формирование выборок номеров на блокировки/ограничения по статусам DUL и синхронизация в downstream-контуры.
* Сопровождение схемы данных: партиционирование, индексация партиций (CONCURRENTLY), SQL/plpgsql-функции под агрегации и обслуживание.

#### Технологии

PHP 8.3, Laravel, Go, PostgreSQL (partitioning, concurrent indexes, plpgsql), Docker, cron, Laravel Scheduler, REST API, CSV import pipelines.
