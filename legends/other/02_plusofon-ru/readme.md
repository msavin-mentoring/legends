## Плюсофон

### Проект: Платформа IP-телефонии и аналитики звонков (Caller ID / антиспам / обогащение CDR)

#### Описание системы

Платформа IP-телефонии, которая обрабатывает и анализирует телефонный трафик, обогащает события звонков справочниками операторов и внутренними источниками, формирует «этикетки» (caller labels) и вычисляет признаки подозрительной активности для дальнейших действий (например, блокировки/ограничения номеров в downstream-системах).

Ключевые термины домена:

* **CDR (Call Detail Record)** — запись детализации звонка (время, номер A, номер B, длительность/статус и т.п.). Это базовый факт для аналитики/биллинга/антиспама.
* **Этикетки звонящих** — пользовательские/системные метки, которые сопоставляются номеру и показываются как идентификация/категория (в т.ч. для входящих/исходящих сценариев).
* **МНП (Mobile Number Portability)** — перенос номера между операторами; требует определения *текущего* оператора номера по справочнику.
* **БДПН** — база перенесённых номеров (используется для определения актуального оператора при МНП).
* **НПДБ / NPDB** — справочник/микросервис с данными по нумерации/операторам (использовался в пайплайне обогащения CDR).
* **МАВ / MAV** — отдельный микросервис/контур, куда прокидывались изменения по номерам и данные для дальнейшей агрегации/использования.
* **КСИМ / XSIM** — контур, где применяются решения по номерам (например, блокировки).
* **ДУЛ / DUL** — внешний API/контур, по которому принимались решения о блокировке/статусах номера и далее синхронизировались действия в XSIM.
* **ЕПВВ** — база/контур исторических сведений/верификации звонков: выгрузка/поиск исторических вызовов по известным спискам номеров.

Основной поток данных (high level):

1. Система вычитывает события звонков из источника телефонии и сохраняет их как CDR в свою PostgreSQL.
2. Отдельным этапом CDR **обогащаются**: определяется оператор по БДПН/НПДБ, дописываются признаки (в т.ч. для последующей агрегации/этикеток).
3. CDR дополнительно обогащаются данными **по агентам**: для номеров A/B определяется текущий «владелец/агент» (через таблицы numbers/agents), чтобы агрегировать статистику по агентам/операторам.
4. Периодические джобы строят агрегаты (например, по часам/суткам): количество вызовов, распределение по операторам, признаки спама (повторяемость/частота), подготовка данных для менеджеров/аналитики.
5. По правилам/статусам (в т.ч. с обращением в DUL API) формируются действия: какие номера блокировать/обновлять и синхронизировать изменения в XSIM и MAV.

#### Архитектура

Ключевые компоненты системы:

* **Backend сервисы**

  * Основной backend на **PHP 8.3 + Laravel** (API, бизнес-логика, планировщик задач, синхронизация контуров).
  * Набор микросервисов (как минимум отдельно выделялись контуры **MAV** и **NPDB**).
  * Высоконагруженные ETL/джобы на **Go** (вытаскивание/перекладка больших объёмов данных, минимизация нагрузки на PostgreSQL).

* **Базы данных**

  * **PostgreSQL** как основная БД.
  * Таблица **CDR партиционирована по дням**.

* **Очереди и фоновые задачи**

  * **cron для Go-джоб**.
  * **Laravel Scheduler** для задач синхронизации и регламентных процедур.
  * Очереди как механизм асинхронной обработки присутствовали; часть пайплайна опиралась на хранение/оркестрацию через БД (по твоему описанию).

* **Кеши**

  * Redis в твоих задачах напрямую не использовался.

* **Внешние интеграции**

  * Интеграции с микросервисами **NPDB / MAV / XSIM / DUL** (часть через API, часть через прямое подключение к БД).
  * Импорт данных из **CSV**, которые выгружал сторонний сервис в **mount-директорию** (дальше — загрузка/фильтрация в таблицы).
  * Хранилище документов в контуре “**Эльза/Эльба**” (по поведению как S3-подобное object storage).
  * Выгрузки/работа с **ЕПВВ** (исторические сведения по звонкам).

* **Взаимодействие**

  * Конвейер: ingestion CDR → запись в PostgreSQL → обогащение операторами (БДПН/НПДБ) → обогащение агентами → агрегации/статистика → синхронизация действий в смежные контуры (MAV/XSIM) и принятие решений по блокировкам (DUL).

#### Нагрузка и данные

* **RPS**

  * Внутрисистемные ограничения по rate limit доходили до порядка **~300 RPS** на интеграции/микросервисы (и иногда требовали временно отключать лимитирование).

* **Количество пользователей**

  * Нет данных (нужно уточнение, если важно для резюме).

* **Объем данных**

  * **CDR ~ 1 000 000 записей/сутки**.
  * **~400 дневных партиций**, в каждой примерно **~1 000 000 записей** ⇒ суммарно порядка **~400 млн CDR** на текущий момент (оценка по указанным цифрам).

* **Размеры ключевых таблиц**

  * Ключевая — **cdr (partitioned, daily)**.
  * Дополнительно: справочники/таблицы по агентам (numbers/agents), таблицы этикеток/номеров, таблицы агрегаций (agg входящая/исходящая этикетка — из задач).

* **Размер очередей**

  * Точных данных нет.

Отдельно по производительности пайплайна ingestion/обработки:

* реализация на PHP работала порядка **~2 часов**,
* оптимизация “на PostgreSQL” давала **~13 минут**, но создавала чрезмерную нагрузку на БД,
* реализация на Go выполняла пайплайн примерно за **~3 минуты** при более контролируемой нагрузке на PostgreSQL.

#### Моя зона ответственности

Зона ответственности — backend-разработка и эксплуатационно-ориентированная оптимизация пайплайнов обработки телеком-данных.

Ключевые области:

* Реализация и оптимизация ingestion-пайплайна CDR: вычитка звонков из базы телефонии, преобразование (time, number_a, number_b и др.), запись в PostgreSQL.
* Проектирование и разработка фоновых задач (Go cron + Laravel Scheduler) для синхронизации состояний между микросервисами/контурами (в т.ч. каскадные удаления/обновления номеров в нескольких таблицах и прокидывание изменений в MAV).
* Обогащение CDR операторами по МНП: построение шага, который после загрузки CDR дописывает актуального оператора (через отдельный микросервис/API), с учетом БДПН/НПДБ.
* Обогащение CDR агентами: определение агента для номеров A/B по таблицам numbers → agents и подготовка данных для дальнейших агрегаций.
* Импорты из CSV (до **~2 млн записей**): загрузка из mount-директории, фильтрация по правилам, запись в целевые таблицы.
* Регламентные аналитические/антиспам-джобы:

  * расчет повторяемости звонков “номер → один и тот же номер” за сутки как сигнал подозрительной активности,
  * агрегации “этикеток” (входящая/исходящая) по часам с группировкой по оператору/агенту,
  * выборка номеров для блокировок по статусам и ответам DUL API с дальнейшей синхронизацией в XSIM.
* Интеграции с хранилищем документов (Эльза/Эльба, S3-подобное) и задачами вокруг ЕПВВ.

Архитектурные решения (фактически применённые в задачах):

* Вынос тяжелых джоб из PHP в Go для сокращения времени обработки и снижения давления на PostgreSQL.
* Партиционирование CDR по дням как базовая стратегия управления объёмом и скоростью запросов.
* Учет rate limit и деградаций интеграций: управление скоростью запросов к микросервисам, возможность временно отключать ограничения при необходимости.

#### Технологии

PHP 8.3, Laravel, Go, PostgreSQL (partitioning), Docker, cron, Laravel Scheduler, REST API, CSV import pipelines.
