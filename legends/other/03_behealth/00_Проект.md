### Проект: BeHealth — мобильная платформа персонального мониторинга здоровья

#### Описание системы

BeHealth — мобильное digital-health приложение для контроля соблюдения рекомендаций врача и подготовки объективного контекста данных пациента перед визитом.

Система не выполняет диагностику и не генерирует медицинские назначения. Основная задача продукта — контроль выполнения уже назначенного лечения (**adherence**) и агрегирование health-данных из внешних источников.

Ключевые сценарии использования:

* ведение расписаний приёма лекарств
* напоминания о приёме препаратов
* импорт данных о состоянии здоровья
* контроль соблюдения рекомендаций врача
* ведение дневников симптомов
* формирование отчётов для врача перед визитом

Ключевой модуль — движок расписаний лекарств с поддержкой сложных схем лечения: ежедневный приём, интервалы (каждые N часов), циклы (например 5 дней приём / 2 дня перерыв) и курсы с датой завершения.

Для хранения расписаний использовалась RRULE-подобная модель повторений: в базе сохранялись правила повторения, а не отдельные записи на каждый день. Это снижало объём данных и упрощало перерасчёт расписаний при изменениях.

Второй важный блок — агрегирование health-метрик из системных источников:

* Apple HealthKit
* Android Health Connect

Импортировались данные по сну, шагам, тренировкам, пульсу и калориям активности. Метрики нормализовались в единую модель наблюдений (`observations`) с идемпотентной записью для защиты от дублей при повторных синхронизациях.

Для проверки выполнения рекомендаций реализован **Adherence Engine**. Рекомендации задавались в виде формальных правил: тип метрики, условие, целевое значение и период (день / неделя / месяц). На основе агрегированных данных система рассчитывала adherence score и присваивала статус выполнения: `fulfilled`, `partially fulfilled`, `missed`.

Пользователи могли вести дневники симптомов (событие, интенсивность, триггеры, заметки). Перед визитом к врачу формировался **Patient Summary** за 30–90 дней с экспортом в PDF: динамика сна, активность, тренировки, соблюдение рекомендаций, история приёма лекарств и симптом-логи.

Поддерживался режим совместного доступа (**caregiver**): владелец профиля мог предоставить доступ родственнику или сиделке для просмотра расписания, отметки приёма лекарств и получения уведомлений о пропущенных дозах.

#### Архитектура

Ключевые компоненты системы:

**Backend сервис**

* REST API для мобильных клиентов
* модуль расписаний лекарств
* модуль импорта health-данных
* rule engine проверки рекомендаций
* сервис уведомлений
* сервис генерации отчётов и PDF

**База данных**

* MySQL для хранения пользовательских и медицинских данных
* ключевые таблицы: `profiles`, `medications`, `medication_schedules`, `observations`, `recommendations`, `symptom_logs`

**Кэш**

* Redis для кеширования агрегированных метрик
* хранение временных вычислений расписаний
* дедупликация отправки уведомлений

**Система уведомлений**

* Local notifications на устройстве (offline-first канал)
* Push-уведомления через FCM / APNS
* quiet hours
* повторные напоминания
* эскалация уведомлений
* идемпотентная отправка
* перерасчёт уведомлений при изменении расписания или часового пояса

**Интеграции**

* Apple HealthKit
* Android Health Connect

Интеграции использовались для импорта метрик здоровья и синхронизации контекста данных с мобильным приложением.

#### Нагрузка и данные

* ~50 000 зарегистрированных пользователей
* ~10 000 daily active users
* средняя API-нагрузка: ~30–50 RPS
* пиковая API-нагрузка: до 120 RPS
* 50–200 наблюдений на пользователя в день
* ~500k–1M записей `observations` в день
* `observations`: 150–250 млн записей при масштабировании
* `medication_schedules`: ~1–2 млн записей
* `symptom_logs`: ~10–20 млн записей
* 2–5 тыс PDF-отчётов в день
* период отчёта: 30–90 дней

#### Моя зона ответственности

* разработка backend-логики платформы
* проектирование REST API для мобильных клиентов
* разработка архитектуры ключевых модулей системы
* участие в рефакторинге и code review

Ключевые реализованные решения:

* универсальная модель повторяющихся расписаний лекарств на основе RRULE-подобных правил
* функционал сбора и нормализации health-данных из Health Connect
* архитектура системы уведомлений с поддержкой offline-first local notifications и push-уведомлений
* RBAC-модель совместного доступа к медицинскому профилю
* модуль визуализации соблюдения рекомендаций (adherence)
* архитектура модулей хранения лекарств и календаря медицинских событий

#### Технологии

PHP 7.3, Laravel 8, MySQL, Redis, Docker, Swagger, JavaScript (Vue)
