# Факап: двойное проведение заказов из-за гонки webhook + retry

## Контекст

Сервис обрабатывает:

1. webhook от платёжки
    
2. retry от очереди при временной ошибке
    
3. fallback-провайдера
    

В определённый момент начали появляться кейсы:

- один и тот же заказ переводился в `paid` дважды
    
- провайдер пополнения вызывался повторно
    
- баланс пополнялся повторно
    

Редкий баг, но финансово чувствительный.

---

# Что произошло

Платёжная система:

- присылала webhook
    
- если не получала 200 OK за N секунд — отправляла повторно
    

Проблема:

1. Первый webhook начал обрабатываться
    
2. Внутри была транзакция + вызов внешнего провайдера
    
3. Ответ 200 задержался
    
4. Платёжка отправила второй webhook
    
5. Второй обработчик прошёл проверку статуса, потому что:
    
    - статус ещё не был закоммичен
        
    - не было SELECT FOR UPDATE
        
    - не было уникального idempotency key
        

В результате — гонка.

---

# Почему это хороший факап

Это:

- не «кто-то нажал не туда»
    
- а архитектурный недочёт
    
- который выявляется только под нагрузкой
    

---

# Как проявилось

- несколько заказов с одинаковым provider_payment_id
    
- двойные вызовы внешнего API
    
- рост компенсаций
    

---

# Как диагностировали

1. Логи показали повторные webhook'и
    
2. В Sentry видно параллельные транзакции
    
3. Нашли отсутствие DB-level защиты
    

---

# Что сделали

1. Ввели idempotency key на provider_payment_id
    
2. Добавили unique index
    
3. Обернули обработку в транзакцию с SELECT FOR UPDATE
    
4. Перенесли вызов провайдера после фиксации статуса
    
5. Добавили метрику повторных webhook’ов
    

---

# Результат

- Полностью исключили двойные проведения
    
- Снизили финансовые риски
    
- Улучшили наблюдаемость
    

---

# Как это рассказывать на собесе

Сильная версия:

> На проде словили редкий race-condition при параллельной обработке webhook’ов. Из-за отсутствия idempotency на уровне БД заказ мог проводиться дважды. Мы воспроизвели сценарий, добавили уникальные ограничения и транзакционную защиту, а также пересмотрели порядок побочных эффектов. После фикса полностью исключили повторные списания.

---

# Почему это полезный факап для легенды

Он показывает:

- понимание транзакций
    
- понимание конкурентности
    
- умение читать логи
    
- знание idempotency
    
- ответственность за финансы
